# Large Goal Reset / Restart / Reassign

Updated: 2026-05-18

Status: implemented in code. `database/migrations/add-goal-id-to-transactions.sql` was applied to the active Supabase database on 2026-05-18.

## Semantics

- Reset keeps the same `student_goals` row. It clears completion state, releases transactions consumed by that goal, removes completion/refund transactions generated by that goal, and recalculates progress from a chosen tracking start date.
- Undo complete is a Reset that keeps the original tracking start date.
- Student-page Reset shows a confirmation modal. The user can choose to restart from today, keep the original tracking start date, or choose a custom tracking start date.
- Restart creates a new active `student_goals` row and preserves the old completed record. When the old goal has `template_id`, the new row copies the latest `goal_templates` fields and event links.
- Reassign is handled from the template assignment UI: stop tracking the current active student goal by setting `is_active=false`, then assign the template again to create a fresh active row.

## Implementation Notes

- New API routes:
  - `POST /api/student-goals/[id]/reset`
  - `POST /api/student-goals/[id]/restart`
  - `POST /api/goal-templates/unassign`
- Compatibility route:
  - `POST /api/student-goals/[id]/reactivate` delegates to reset with `tracking_started_at_mode: "today"`.
- New migration:
  - `database/migrations/add-goal-id-to-transactions.sql`
- Diagnostic SQL:
  - `database/queries/goal-progress-vs-reward-stats-diff.sql`
- `transactions.goal_id` is available in the active database. Reset cleanup uses it for precise deletion of goal-generated transactions. The API still keeps a constrained legacy fallback for older or freshly initialized environments where the migration has not yet been applied.
